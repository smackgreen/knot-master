# Supabase Storage Setup — Knot To It

This guide documents every Supabase Storage bucket the application requires or
recommends, along with privacy settings, allowed MIME types, size limits, and
the Row-Level Security (RLS) policies for each.

Create buckets from the Supabase Dashboard → **Storage** → **New bucket**, or
via the one-shot SQL at the end of this file.

---

## Bucket Overview

| # | Bucket | Privacy | Status | Primary Consumer |
|---|--------|---------|--------|------------------|
| 1 | `documents` | Private | **Required** | `storageService.ts`, `documentService.ts` |

## 1. `documents` — **Required** ✅

**Actively used** by the application today.

| Property | Value |
|---|---|
| Bucket name | `documents` |
| Privacy | **Private** (signed URLs only) |
| File size limit | **50 MB** |
| Allowed MIME types | `application/pdf`, `image/png`, `image/jpeg`, `image/webp`, `application/msword`, `application/vnd.openxmlformats-officedocument.wordprocessingml.document`, `application/vnd.ms-excel`, `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`, `text/plain`, `text/csv` |
| Path convention | `{user_id}/{uuid}-{filename}` |

### Source files

- [`storageService.ts`](src/services/storageService.ts) — `uploadFile()`, `getSignedUrl()`, `downloadFile()`, `deleteFile()` all reference `DOCUMENTS_BUCKET = 'documents'`
- [`documentService.ts`](src/services/documentService.ts) — `createDocument()` uploads via `uploadFile()` and verifies bucket existence with `supabase.storage.listBuckets()`
- [`UppyUploader.tsx`](src/components/documents/UppyUploader.tsx) — direct upload fallback uses `supabase.storage.from(DOCUMENTS_BUCKET).upload()`
- [`UploadDocumentDialog.tsx`](src/components/documents/UploadDocumentDialog.tsx) — wraps UppyUploader for document uploads
- [`UploadDocumentForSignatureDialog.tsx`](src/components/signatures/UploadDocumentForSignatureDialog.tsx) — uploads documents for signature workflows
- [`SignDocumentDialog.tsx`](src/components/documents/SignDocumentDialog.tsx) — retrieves signed URLs for PDF viewing
- [`SignDocument.tsx`](src/pages/SignDocument.tsx) — public signing page uses `getSignedUrl()`
- [`DocumentsList.tsx`](src/components/documents/DocumentsList.tsx) — displays documents with signed URLs

### Access pattern

All reads use signed URLs generated by `getSignedUrl()` with a **1-hour expiry**.
No public access is granted.

### RLS policies (storage.objects)

```sql
-- SELECT: owner can read any file under their own user_id prefix
CREATE POLICY "documents_owner_select" ON storage.objects
  FOR SELECT TO authenticated
  USING (bucket_id = 'documents' AND (storage.foldername(name))[1] = auth.uid()::text);

-- INSERT: owner can write only under their own prefix
CREATE POLICY "documents_owner_insert" ON storage.objects
  FOR INSERT TO authenticated
  WITH CHECK (bucket_id = 'documents' AND (storage.foldername(name))[1] = auth.uid()::text);

-- UPDATE: owner only
CREATE POLICY "documents_owner_update" ON storage.objects
  FOR UPDATE TO authenticated
  USING (bucket_id = 'documents' AND (storage.foldername(name))[1] = auth.uid()::text);

-- DELETE: owner only
CREATE POLICY "documents_owner_delete" ON storage.objects
  FOR DELETE TO authenticated
  USING (bucket_id = 'documents' AND (storage.foldername(name))[1] = auth.uid()::text);
```

> **Note:** The public signing page (`/sign/{token}`) accesses documents via
> authenticated server-side calls. If unauthenticated access is needed, add a
> separate policy scoped to a service role or use an Edge Function to proxy
> signed URLs.

## Troubleshooting: 403 Unauthorized on Document Upload

### Symptoms

- Uploading a document via the Uppy uploader fails with **403 Unauthorized**
- Browser console shows: `new row violates row-level security policy`
- The `POST` to `/storage/v1/object/documents/...` returns `403`

### Root Cause

The `storage.objects` table has Row-Level Security (RLS) enabled by default in
Supabase. If no INSERT policy exists for the `documents` bucket, all INSERT
operations are denied — even for authenticated users. The policies documented
above in the **One-shot Creation SQL** section must be explicitly executed on
the database.

### Fix

Run the dedicated fix script [`fix_documents_rls_policies.sql`](fix_documents_rls_policies.sql)
in the **Supabase SQL Editor**:

1. Go to [Supabase SQL Editor](https://app.supabase.com/project/_/sql)
2. Paste the contents of `fix_documents_rls_policies.sql`
3. Click **Run**

This script is idempotent — it drops existing `documents_*` policies first, then
recreates them. It also creates the bucket if it doesn't exist.

### Diagnostic Queries

**Check if policies exist:**

```sql
SELECT policyname, cmd, permissive, roles
FROM pg_policies
WHERE schemaname = 'storage'
  AND tablename = 'objects'
  AND policyname LIKE 'documents%';
```

Expected: 4 rows (SELECT, INSERT, UPDATE, DELETE).

**Check if the bucket exists:**

```sql
SELECT * FROM storage.buckets WHERE id = 'documents';
```

Expected: 1 row with `public = false`.

**Test upload from the browser console** (replace `YOUR_USER_ID`):

```js
const { data, error } = await supabase.storage
  .from('documents')
  .upload('YOUR_USER_ID/test.txt', new Blob(['test'], { type: 'text/plain' }));
console.log(data, error);
```

Expected: `data` object with the file path, `error` is `null`.

### How the RLS Policy Restricts Uploads to User-Owned Folders

The INSERT policy uses this check:

```sql
(storage.foldername(name))[1] = auth.uid()::text
```

- `name` is the full file path in `storage.objects` (e.g., `a1b2c3d4/ef56-gh78-file.pdf`)
- `storage.foldername(name)` splits by `/` and returns a text array: `['a1b2c3d4', 'ef56-gh78-file.pdf']`
- `[1]` extracts the first folder: `'a1b2c3d4'` (the user ID)
- `auth.uid()::text` returns the authenticated user's UUID as text
- If they match → INSERT allowed; otherwise → **403 denied**

This means a user with ID `a1b2c3d4` can upload to `a1b2c3d4/anything.pdf`
but **cannot** upload to `z9y8x7w6/anything.pdf` (another user's folder).

### Common Pitfalls

| Issue | Cause | Fix |
|-------|-------|-----|
| Policies exist but upload still fails | User session expired; `auth.uid()` returns NULL | Re-authenticate in the app |
| Bucket not found (404) | Bucket was never created | Run the bucket creation SQL from the fix script |
| Works locally but not in production | Policies were only applied to the local instance | Run the SQL against the production Supabase project |
| `storage.foldername` returns empty | Path doesn't start with `{user_id}/` | Verify upload path in [`UppyUploader.tsx`](src/components/documents/UppyUploader.tsx:167) |
